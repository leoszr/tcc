# H02.4 — Service: checar conflito por data (PENDING/APPROVED)

Objetivo:
Documentar a lógica que impede criação de nova reserva para uma `data_evento` que já possua um registro com status `PENDING` ou `APPROVED`.

Entradas / Saídas:

- Entrada: `data_evento` (date)
- Saída: `ok` booleano ou erro com código HTTP 409 e payload `{ code: "DATE_CONFLICT", message: "Já existe uma solicitação para esta data" }`.

Regras:

- Consultar repositório por registros com a mesma `data_evento` e status `PENDING` ou `APPROVED`.
- Se encontrado, bloquear criação com erro 409.

Critérios de Aceite:

- Documento `docs/services/checar_conflito.md` com a lógica e exemplos.
- Testes unitários (H09.1) devem cobrir cenários: sem conflito (ok) e com conflito (erro 409).

Dependências:

- H02.2 (repository) para a consulta; H02.1 (model).

Estimativa: 0.5d (4 horas)

Checklist de QA:

- Preparar cenário no banco com uma reserva PENDING e tentar criar outra para a mesma data; verificar retorno 409.
